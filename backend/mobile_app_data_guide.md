# دليل بيانات وتصميم تطبيق الموبايل للطالب (Mobile App Data & UI/UX Guide)

بناءً على تحليل الـ Backend الخاص بنظام الحضور والانصراف، قمت بتجميع كافة البيانات والنهايات الطرفية (Endpoints) التي تهم تطبيق الموبايل الخاص بالطالب. هذا الملف موجه بشكل أساسي لمصمم الـ UI/UX ومطوري الـ Mobile لفهم شكل البيانات المتاحة وتصميم الواجهات بناءً عليها.

---

## 1. بيانات الطالب (Student Profile)
هذه البيانات تستخدم في صفحة الملف الشخصي (Profile) وأيضاً لعرض اسم الطالب في الصفحة الرئيسية.

* **الـ Endpoint المستخدمة:** `GET /api/users/me`
* **البيانات الراجعة (Response):**
```json
{
  "success": true,
  "data": {
    "_id": "60d0fe4f5311236168a109ca",
    "email": "student@example.com",
    "name": "أحمد محمد",
    "role": "student",
    "studentId": "20230001",
    "phone": "01000000000",
    "isActive": true,
    "academicInfo": {
      "department": { "_id": "...", "name": "علوم الحاسب" },
      "level": 3,
      "enrolledCourses": [ 
        { "_id": "...", "name": "هياكل البيانات", "code": "CS201" } 
      ]
    },
    "device": {
      "macAddress": "00:11:22:33:44:55",
      "deviceId": "device-uuid",
      "deviceModel": "iPhone 13",
      "isApproved": true
    }
  }
}
```
💡 **اقتراح لـ UI/UX:** 
- تصميم صفحة "الملف الشخصي" تعرض صورة الطالب (إن وجدت مستقبلاً)، الاسم، الرقم الجامعي، المستوى الدراسي، والقسم.
- جزء خاص بـ "الجهاز المسجل" يوضح ما إذا كان جهازه الحالي معتمد لتسجيل الحضور، مع توفير زر "طلب تغيير الجهاز" (Request Device Change).

## 2. جدول المحاضرات (My Schedule)
لعرض جدول محاضرات الطالب طوال الأسبوع أو محاضرات اليوم فقط. الـ Backend يرجع الجدول مقسم حسب أيام الأسبوع (من 0 الأحد إلى 6 السبت).

* **الـ Endpoint المستخدمة:** `GET /api/lectures/my-schedule`
* **البيانات الراجعة (Response):**
```json
{
  "success": true,
  "data": {
    "0": [ // يوم الأحد
      {
        "_id": "...",
        "course": { "name": "قواعد البيانات", "code": "CS301" },
        "hall": { "name": "مدرج 1", "building": "مبنى أ" },
        "doctor": { "name": { "first": "علي", "last": "حسن" } },
        "startTime": "10:00",
        "endTime": "12:00",
        "dayOfWeek": 0
      }
    ],
    "1": [], // الإثنين
    "2": [], // الثلاثاء
    // ... باقي الأيام
  }
}
```
💡 **اقتراح لـ UI/UX:** 
- **بالنسبة للصفحة الرئيسية (Home):** يتم أخذ مصفوفة "اليوم الحالي" فقط (مثلاً اليوم الأحد يتم عرض الـ Array رقم 0) وعرضها كبطاقات (Cards) أفقية أو عمودية تحت عنوان "محاضرات اليوم".
- **صفحة الجدول (Schedule):** عرض أيام الأسبوع كـ Tabs، وعند الضغط على كل يوم يتم عرض قائمة المحاضرات (Timeline view).

## 3. حالة الحضور المباشرة (Live Lecture Status)
تستخدم هذه الـ Endpoint للتحقق مما إذا كان الطالب متواجداً حالياً داخل محاضرة وتُحدث حالة حضوره (يتم عمل Polling لها كل 30 ثانية مثلاً).

* **الـ Endpoint المستخدمة:** `GET /api/attendance/my/status`
* **البيانات الراجعة (Response):**
```json
{
  "success": true,
  "data": {
    "inLecture": true, // هل هو في محاضرة حالياً؟
    "session": {
      "hallName": "مدرج 1",
      "courseName": "هياكل البيانات",
      "courseCode": "CS201",
      "doctorName": "د. أحمد",
      "startTime": "09:00",
      "endTime": "11:00",
      "connectedAt": "2024-03-21T09:05:00.000Z", // وقت الدخول الفعلي للمدرج
      "attendanceStatus": "in-progress", // الحالة: in-progress, present, absent, late, excused
      "presenceTimeMinutes": 45 // عدد الدقائق التي قضاها الطالب حتى الآن داخل المدرج
    }
  }
}
```
💡 **اقتراح لـ UI/UX:** 
- تصميم **Live Activity Banner أو Widget** يظهر أعلى الصفحة الرئيسية *فقط* عندما يكون الطالب داخل المحاضرة (`"inLecture": true`). 
- يجب أن يعرض الـ Widget: اسم المادة، عداد الوقت (للدقائق التي حضرها `presenceTimeMinutes`)، وحالة الحضور الحالية بلون مميز (أخضر=مقبول، أصفر=متأخر).

## 4. ملخص حضور مادة معينة (Course Attendance Summary)
تستخدم عند دخول الطالب لتفاصيل مادة معينة ليرى نسبة غيابه وهل هو في خطر الحرمان أم لا.

* **الـ Endpoint المستخدمة:** `GET /api/attendance/my/course/:courseId`
* **البيانات الراجعة (Response):**
```json
{
  "success": true,
  "data": {
    "course": {
      "name": "هياكل البيانات",
      "code": "CS201"
    },
    "summary": {
      "totalLectures": 10,       // إجمالي المحاضرات التي انعقدت
      "presentLectures": 8,      // مرات الحضور
      "absentLectures": 2,       // مرات الغياب
      "attendancePercentage": 80 // نسبة الحضور الكلية 80%
    },
    "records": [ // السجل التاريخي للحضور
      {
         "_id": "...",
         "date": "2024-03-21T00:00:00.000Z",
         "status": "present", // أو absent أو late
         "lecture": { "startTime": "09:00", "endTime": "11:00", "dayOfWeek": 2 }
      }
    ]
  }
}
```
💡 **اقتراح لـ UI/UX:** 
- تصميم **Progress Bar أو Circular Chart** يوضح نسبة الحضور (`attendancePercentage`).
- إذا كانت النسبة أقل من **75%**، يظهر تنبيه باللون الأحمر (Alert/Warning) للطالب بأنه معرض للحرمان من المادة.
- عرض قائمة `records` كـ "سجل الحضور التاريخي" بوضع أيقونة (صح ✅ / خطأ ❌) بجانب تاريخ كل محاضرة.

---

## 5. السجل الكامل للحضور (All Attendance Records)
يمكن استخدامه لعرض صفحة كاملة خاصة بسجلات الحضور مرتبة زمنيًا (أحدث وأقدم).

* **الـ Endpoint المستخدمة:** `GET /api/attendance/my`
* **البيانات الراجعة (Response):** Pagination list ترجع بيانات مشابهة لبيانات الـ records السابقة.

## مقترحات إضافية للتطبيق (Suggestions & Best Practices):

1. **الصفحة الرئيسية (Home Dashboard):**
   - **الترحيب المخصص:** "مرحباً يا [اسم الطالب]، لديك اليوم [عدد] محاضرات".
   - **بطاقة الحضور المباشر (Live Status):** تظهر فقط وقت تواجد الطالب في المدرج (باللون الأخضر المضيء - Pulse Effect).
   - **الجدول السريع:** عرض المحاضرة "القادمة" (Up Next) مع تحديد المكان والزمان ومؤقت للعد التنازلي لبدايتها.

2. **نظام الإشعارات (Notifications):**
   - يفضل تصميم شاشات للإشعارات المنبثقة: 
     - *"تم تسجيل دخولك بنجاح في محاضرة هياكل البيانات"*.
     - *"تم احتساب غيابك لليوم في مادة الشبكات لتخطي نسبة الغياب المسموحة"*.

3. **شاشة تغيير الهاتف (Device Change Request):**
   - بناءً على نظام الباك اند، يتم تسجيل الماك أدرس (MAC Address). إذا اشترى الطالب هاتفاً جديداً يحتاج إلى طلب تغيير. 
   - يمكن إضافة شاشة في الإعدادات تحتوي على Form لتقديم عذر (مثلاً: الهاتف القديم ضاع) ليوافق عليه الأدمن، استناداً لـ Endpoint: `POST /api/students/request-device-change`
